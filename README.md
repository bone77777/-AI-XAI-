# 処方的AI × 説明可能AI × 深層強化学習による SOC意思決定エンジン (PoC)

> セキュリティオペレーションセンター (SOC) におけるアラート応答を、
> **(1) なぜそう判断したかを示す**（XAI）／
> **(2) 業務影響を加味して最適行動を選ぶ**（処方的AI）／
> **(3) 複数ステップにわたり状況に応じて再評価する**（強化学習）
> という3層で実装した、概念実証 (Proof of Concept) のリポジトリです。

> [!IMPORTANT]
> 本リポジトリは **原理レベルの再現実験** であり、本番SOC環境への適用を想定したものではありません。報酬関数の値、特徴量の重み、行動の3カテゴリ化は、いずれも筆者が学習目的で設計した「現実のSOCの単純化モデル」です。詳しくは末尾の「制限事項」を必ずお読みください。

## 背景 — なぜこの3層を組み合わせるのか

現代のSOCには、性質が異なる3つの問題が同時に存在しています。

1. **アラート過多と判断の自動化要求**
   SIEMが上げてくるアラートは1日数千〜数万件規模で、すべてを人手で見るのは非現実的です。何らかの自動応答が必要になります。

2. **ブラックボックスAIへの正当な不信**
   一方で、自動応答が「なぜブロックしたのか」を説明できなければ、誤検知時の事後検証も、運用ルールへのフィードバックもできません。GDPR Art.22 の自動意思決定への異議申立権や、各種AIガバナンス枠組み（NIST AI RMF, ISO/IEC 42001, EU AI Act）も、説明可能性を要件として置いています。

3. **業務影響を無視した自動防御の危険**
   「疑わしいから遮断」を機械的に実行すると、決算期の経理部の通信を遮断するなど、防御がそのままビジネス被害を生む事故が起きます。攻撃の被害よりも、対応そのものの被害が大きい状況は珍しくありません。

この3つを別々に解こうとすると、それぞれ対症療法になります。本リポジトリでは、

- **XAI** で「なぜそのリスク評価か」を示し、
- **処方的AI (Prescriptive AI)** で「どの行動が最も有益か」を効用計算で選び、
- **強化学習 (DQN)** で「1回の判断で終わらず、状況の推移を観察しながら段階的に対応する」

という3層を一つの意思決定パイプラインに統合する形で実装しました。

## v1: 単一ステップ版 (`Prescriptive AI×XAI.py`)

### 何をするか

ログデータ（通信量、TIレピュテーション、署名の有効性、プロセスの正規性）から、

1. **線形重み** で各特徴量の寄与度を計算しリスクスコアを算出（XAI）
2. **3つのアクション**（`ISOLATE` / `THROTTLE` / `MONITOR`）について、
   `効用 = 防御で減らせるリスク − 業務停止コスト` を計算（処方的AI）
3. **業務コンテキスト**（決算期かどうか）で資産価値を動的に重み付け
4. 効用が最大の行動を「処方箋」として提示し、その理由を出力

### 設計の要点

- **線形モデル** を採用しているので、各特徴量の寄与度がそのまま解釈可能です。SHAPやLIMEのような事後説明手法を別途かぶせる必要がありません。
- **コストは固定スカラー** で表現しています。これは「業務影響をスコアに乗せる」ことの最小実装で、本来は資産DB・MTTR・SLAペナルティなどから動的に計算されるべき値です。
- **決算期フラグ** は「業務コンテキストを意思決定に乗せる」ことを示すための1ビットの例示です。実運用では時期・部門・取引相手・規制要件など複数次元で表現されるはずです。

### 入出力例

入力（コード末尾の実行ブロック）:
```python
log_data = {
    "traffic_volume": 1.0,     # 大量通信
    "ti_reputation": 0.5,      # サブネットがTIにヒット
    "valid_signature": 1.0,    # 署名あり（白要素）
    "internal_process": 1.0    # 正規プロセス（白要素）
}
context = {"is_closing_period": True}  # 決算期
```

出力（要約）:
```
【XAI：判定根拠の可視化】
   traffic_volume  : +0.70
   ti_reputation   : +0.20
   valid_signature : -0.80
   internal_process: -0.50
  => 最終リスク確信度: 0.00%

【最終処方箋】
推奨アクション: 監視のみ
処方理由: リスク(0.0%)に対して、業務停止損失を最小化しつつ ...
```

「黒要素はあるが、白要素のほうが強い」というケースで、決算期の業務影響を考慮し、自動遮断ではなく監視継続を選ぶ判断が再現できています。

## v2: マルチステップ + DQN拡張版 (`強化学習追加バージョン`)

v1の最大の限界は、**1回の判断で対応が終わってしまう**ことです。実際のインシデント対応は「最初は監視 → 怪しさが増したので帯域制限 → やはり危険なので遮断」「最初に遮断したが、誤検知だったので解除」のように、状況の推移を見ながら段階的に進みます。

v2はこれを **マルチステップMDP（マルコフ決定過程）** として再定義し、DQN で最適方策を学習させたものです。

### v1からの主な拡張

| 項目 | v1 | v2 |
|---|---|---|
| 意思決定の長さ | 1ステップ | 最大5ステップ |
| 状態表現 | 4次元 | **12次元**（履歴・時間帯・健全度を含む） |
| 行動選択 | 効用関数の直接計算 | DQNが学習したQ値による選択 |
| エスカレーション | 概念なし | 段階的引き上げ／引き下げを報酬で誘導 |
| 学習 | なし（ルールベース） | Experience Replay + Target Network |
| 業務影響 | 決算期フラグのみ | 決算期 × 時間帯（業務時間/夜間/休日） |

### 状態空間（12次元）の内訳

| 次元 | 内容 |
|---:|---|
| 1 | リスクスコア（XAIモジュールの出力） |
| 2 | 決算期フラグ |
| 3-5 | 時間帯（業務時間 / 夜間 / 休日のone-hot） |
| 6 | システム健全度（0〜1） |
| 7 | エスカレーション回数（正規化） |
| 8 | 現在のステップ番号（正規化） |
| 9-11 | 過去3ステップの行動履歴 |
| 12 | 観測可能な脅威強度（IDS/IPS出力に相当、ノイズあり） |

### 報酬関数の構造

報酬は4成分の和として設計されています:

```
R = R_defense + R_health + R_disruption + R_escalation
```

- `R_defense`: 高脅威時に十分な防御を行えば加点、見逃せば大きく減点。低脅威時は静観に加点、過剰反応に減点
- `R_health`: ダメージを受けると減点、健全度が高いと小ボーナス
- `R_disruption`: 業務中断に応じた減点（決算期×業務時間で増幅）
- `R_escalation`: 高脅威時のエスカレーション／低脅威時のデエスカレーションに加点、低脅威時の不要なエスカレーションには減点

これにより、エージェントは「**いつ攻めて、いつ引くか**」の段階的判断を獲得することを期待しています。

### DQNの実装上の選択

- **Experience Replay**（容量50,000）で時間相関を切る
- **Target Network**（200ステップごとに同期）で過大評価を抑制
- **Smooth L1 Loss (Huber)** で外れ値報酬の影響を緩和
- **勾配クリッピング**（ノルム10.0）で発散防止
- **LayerNorm** を中間層に挿入し、状態スケールの差による不安定化を抑制
- **ε-greedy**: ε = 1.0 → 0.05 まで指数減衰

### 学習済み方策の確認

`_print_policy_analysis()` メソッドが、代表シナリオでQ値テーブルを出力します。例:

```
シナリオ                          MONITOR  THROTTLE  ISOLATE   初手
─────────────────────────────────────────────────────────────
LOW × 通常期 × 業務時間            +X.X    +X.X      +X.X     監視のみ
HIGH × 決算期 × 業務時間           +X.X    +X.X      +X.X     帯域制限
CRITICAL × 通常期 × 業務時間       +X.X    +X.X      +X.X     完全遮断
```

「同じHIGHリスクでも、決算期×業務時間ではISOLATEではなくTHROTTLEを選ぶ」のような、業務文脈に依存した方策が学習されているかを定性的に確認できます（**学習結果は乱数シード・エピソード数で揺らぎますので、再現実験として読んでください**）。

## 設計上の判断とその理由

### なぜXAIに線形モデルを使ったか

ニューラルネットワークでリスク算出すれば精度は上がりますが、SHAP等の事後説明手法を挟む必要があり、**説明そのものが近似値になります**。本PoCの目的は「説明可能性が運用フェーズで何を意味するか」を体感することなので、線形モデルで「重み × 値 = 寄与度」が定義通り出るシンプルな構造を選びました。本番環境では、上流のリスク算出には別モデルを置きつつ、その出力を線形モデルで再構成する蒸留型XAIにする選択肢があります。

### なぜ処方的AIを噛ませるのか

リスク算出だけでは「何をすべきか」は決まりません。リスクスコア80%でも、対応コストがそれ以上なら何もしないのが正解になりうるからです。**意思決定問題はリスク予測問題と異なる**という認識を、コードで明示するために独立した層として実装しました。

### なぜDQNを使ったか（PPOやポリシー勾配ではなく）

行動空間が離散3個と小さく、状態空間も12次元と低次元なので、価値ベースのDQNで十分扱えます。PPO等のポリシー勾配系を持ち出すと、ハイパーパラメータが増えて再現性の検証コストが上がるため、最小限の実装に留めました。

### なぜマルチステップにしたか

単一ステップで「最適行動」を選ぶ問題と、複数ステップで「対応シーケンス」を選ぶ問題は、解の構造が異なります。前者は静的な効用最大化、後者は将来の状態に対する期待値の最大化です。**実際のインシデント対応は後者**なので、後者として定式化することそのものに学習価値があると考えました。

### 報酬関数を4成分に分けた理由

防御・健全度・業務中断・エスカレーションを単一の重み付き和にすると、各成分の貢献が見えなくなり、報酬ハッキングの原因を特定できなくなります。明示的に分割することで「なぜこの方策に収束したか」を分析しやすくしました。

## 制限事項

本リポジトリは原理確認用のPoCであり、以下の点は明示的に**実現していない／本番環境への外挿は不可能**です。

1. **特徴量とその重みは仮想設定**
   実際のSOCログには数百〜数千の特徴量があり、その重みは大量のラベル付きデータからの学習で決まります。本コードの4特徴量・固定重みは「XAIの寄与度可視化が定義通り動くか」を見る最小例です。

2. **報酬関数の数値は筆者が設計した値**
   `defense × 8.0`、`damage × 20.0` などの定数は、PoCで「学習がそれらしく進む」ように選んだものです。実運用では業務インパクトの実測値・SLA違反コスト・規制制裁金の期待値などから別途設計されるべきです。

3. **行動空間が3カテゴリのみ**
   実際のSOCには「ユーザー認証強制再要求」「特定ホストのみブロック」「IRチームへエスカレーション」など多数の行動があります。3カテゴリ化は意思決定の構造を見せるための単純化です。

4. **「真の脅威強度」を環境が知っている前提**
   学習用にシミュレータが `threat_intensity` を持っていますが、実際は誰も真値を知りません。本番ではブラインド評価が必須です。

5. **敵対的摂動への耐性は未検証**
   攻撃者がログ特徴量を意図的に操作してリスクスコアを下げるシナリオ（adversarial example）は本PoCの範囲外です。Goodfellow らの FGSM 系統の摂動に対する頑健性は別途評価が必要です。

6. **オフライン強化学習・分布シフトへの対応なし**
   本実装はオンライン学習を前提としていますが、SOC現場ではオフラインRL（過去ログだけからの方策学習）と本番環境との分布シフトへの対応が必要になります。Conservative Q-Learning (CQL) などの手法を別途検討する必要があります。

7. **多目的最適化として未定式化**
   防御・可用性・コスト・規制要件は本来は多目的最適化問題です。本実装は単一スカラー報酬への重み付き和でこれを近似していますが、Pareto最適性の議論は省いています。

## 今後の拡張余地

- **オフラインRL化**: 既存SOCログからの方策学習（CQL, IQL）
- **マルチエージェント化**: 部門ごとに異なる方策を学習し、CISOエージェントが調停する階層構造
- **敵対的頑健性の評価**: ログ改ざんに対する方策の安定性
- **LLM統合**: 自然言語でのインシデント説明・推奨行動の整形を上流に追加
- **本番ログ形式（ECS, OCSF）への対応**: 現状のディクショナリ表現から標準スキーマへの移行

## 著者ノート

このPoCは、「AIによる自動防御」と「説明責任」と「業務影響への配慮」が同時に成立するパイプラインがどう書けるかを、自分の手で動かして確かめるために作りました。8,000エピソードの学習で「決算期×業務時間ではISOLATEを我慢する」方策がそれっぽく出てきたとき、報酬設計が方策に直接効いている感覚を初めて掴めました。コードとしては最小ですが、設計の判断ポイントは可能な限り明示するように努めました。誤りや改善提案があれば Issue でお知らせいただけると嬉しいです。
